# GitHub Issue ワークフロー

Issue を作業キューとして使用する。

## ブランチ戦略

- `main` — 安定ブランチ、保護済み、直接プッシュ禁止
- `feature/issue-<番号>-<短い名前>` — 実装 Issue 1 件
- `fix/issue-<番号>-<短い名前>` — 修正 Issue 1 件
- `hotfix/issue-<番号>-<短い名前>` — `main` からの緊急修正

マージ後もブランチは削除しない。

## フロー

1. Issue を作成または選択する。
2. `main` からブランチを切る。
3. PR を 1 Issue に絞る。
4. `main` に対して PR を開く。
5. PR 本文で Issue を参照する。
6. CI 通過後、デフォルトで Squash マージする。
7. マージ後もブランチを残す。

## マージポリシー

インテグレーションブランチの履歴を Issue/PR 単位で読めるよう、デフォルトで Squash マージを使用する。内部のコミット履歴が保持する価値があるときのみ通常のマージを使用する。

意図的な変更がない限り、Rebase マージは使用しない。

## ブランチ保護ルール

`main` を保護する:

- マージ前に PR を必須にする
- ステータスチェックの通過を必須にする
- ブランチを最新状態に保つことを必須にする
- 強制プッシュをブロックする
- ブランチ削除をブロックする

## ローカルブランチコマンド

```bash
git switch main
git switch -c feature/issue-2-repo-bootstrap
```
